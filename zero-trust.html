Section 4

Identity & Device Trust — The Control Plane

Identity is the new perimeter in Zero Trust. Every access request must be backed by a verifiable identity claim from a trusted IdP and a device posture assessment from MDM. Both must pass before the Policy Engine evaluates the request.

Identity verification layers
What the IdP and MFA system evaluate
Primary credential: Password, passkey, or FIDO2 hardware token. Passwordless (FIDO2/WebAuthn) is the strongest option — credentials are device-bound and cannot be phished or replayed by an attacker.
Multi-factor authentication: FIDO2 (phishing-resistant, highest tier), TOTP authenticator app (strong, widely deployed), SMS OTP (weakest — susceptible to SIM swap, use only as fallback).
Conditional access signals: Sign-in location (geolocation, IP reputation), time of day vs. normal login hours, new device flag, impossible travel detection (London → Singapore in 30 minutes).
Federation protocols: SAML 2.0 for enterprise SSO, OIDC/OAuth 2.0 for modern app authentication, SCIM for automated user provisioning across connected SaaS platforms.
Privileged identity: PAM enforces just-in-time elevation — admin rights granted for a defined time window then auto-revoked. No standing admin access. Full session recording. Break-glass procedure for emergencies.
Device posture assessment
What MDM and device trust evaluate before granting access
OS patch level: Device must have applied security patches within the SLA window (e.g. critical patches within 24h). Unpatched devices are quarantined to a remediation VLAN — read-only access only.
Disk encryption: BitLocker (Windows), FileVault (macOS), or equivalent must be enabled and actively encrypting. Recovery keys escrowed to MDM. Unencrypted devices denied all corporate access.
EDR agent health: Endpoint Detection and Response agent must be installed, running, and connected to the management console. Tampered or disabled EDR agent = non-compliant device.
Device certificate: Corporate PKI issues a device certificate that cryptographically proves this specific device is corporate-enrolled. Certificate-based device authentication replaces IP-based trust entirely.
BYOD / unmanaged: Devices without MDM enrolment are assigned the "unmanaged" posture tier — restricted to browser-only access via Remote Browser Isolation. No data download, no native app access, no sync.

Section 5

Traditional Perimeter vs Zero Trust

Understanding what Zero Trust replaces — and why — is essential for communicating its value. The traditional castle-and-moat model fails in a world of cloud, remote work, and insider threats.

ConceptTraditional perimeter modelZero Trust model
Trust modelImplicit — inside the network = trustedExplicit — every request verified regardless of location
Network accessVPN grants broad network-level access to entire subnetZTNA grants app-level access only to specific application
Lateral movementCompromised device can reach all systems in the same subnetMicrosegmentation limits blast radius to one workload
Identity verificationVerified once at VPN login; session trusted indefinitelyContinuously re-evaluated; trust score updated in real time
Device postureAny device inside the network is trustedDevice must pass compliance check before every session
Remote accessFull VPN tunnel to corporate network — large attack surfaceApplication-specific tunnel — internal network never exposed
Insider threatInsider with network access can reach most systemsLeast-privilege scoping limits insider access to job function
Cloud resourcesBackhauled through VPN — high latency, poor experienceDirect access via ZTNA proxy at nearest cloud PoP
Breach impactAttacker with foothold can move freely east-westAttacker contained to one microsegment; C2 traffic detected immediately
VisibilityLimited — most internal traffic is unmonitoredFull — every access request logged with identity and device context
