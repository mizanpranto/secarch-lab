Section 2

Email Authentication — SPF · DKIM · DMARC

The three complementary DNS-based protocols that collectively verify a sender's identity and ensure message integrity. All three must be configured correctly to prevent domain spoofing and impersonation attacks.

SPF
Sender Policy Framework · RFC 7208
Validates that the IP address used to send an email is authorised for the sender's claimed domain. The sending domain publishes a list of permitted IP addresses in a DNS TXT record.
1
Receiving server extracts the MAIL FROM domain from the SMTP envelope
2
Queries DNS: example.com TXT "v=spf1 include:..."
3
Checks if the sending server's IP is listed in the SPF record
4
Returns Pass / Fail / SoftFail / Neutral to the DMARC evaluator
✓ Pass: sending IP is authorised · ✗ Fail: DMARC policy applies
Limitation: SPF only checks the envelope MAIL FROM, not the header From address visible to users. Does not prevent display-name spoofing.
DKIM
DomainKeys Identified Mail · RFC 6376
Adds a cryptographic signature to outgoing email headers, proving that the message originated from an authorised server and was not modified in transit.
1
Sending server hashes selected headers + part of body using a private key
2
Signature inserted as a DKIM-Signature: header in the email
3
Receiving server retrieves the public key from DNS: selector._domainkey.example.com
4
Validates the signature — proves identity and that content was not altered
✓ Valid: message integrity confirmed · ✗ Invalid: DMARC policy applies
Key rotation: DKIM selectors should be rotated every 6–12 months. Compromised private keys allow attackers to forge signed messages.
DMARC
Domain-based Message Authentication · RFC 7489
Ties SPF and DKIM together and tells receiving servers what to do when authentication fails. Enables domain owners to receive aggregate and forensic reports about their email streams.
1
Receiving server evaluates SPF and DKIM results for alignment with the header From domain
2
Looks up DMARC policy: _dmarc.example.com TXT
3
Applies the p= policy to failed messages: none / quarantine / reject
4
Sends aggregate reports (rua=) and forensic reports (ruf=) back to the domain owner
p=reject is the strongest setting — eliminates spoofed domain email entirely
Recommended path: Start with p=none (monitoring) → p=quarantine (gradual) → p=reject (full enforcement) after analysing reports.
DMARC policy reference
Policy (p=)SPF/DKIM fails — what happensReports sentUse when
noneMessage delivered normally, no enforcement action takenYes — aggregate & forensic reports collectedStarting out. Monitoring phase — do not enforce until reports are clean

Log Types & SIEM Integration

The email gateway is a high-fidelity log source for the SOC. Every processed message generates a structured log record forwarded to the SIEM in real time. These logs are correlated with endpoint and identity telemetry to detect multi-stage attacks.
